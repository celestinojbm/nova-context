## Preferred future sale posture (operator-attested strategy)

1. Buyer-controlled accounts are created (GitHub org, providers, registrar).
2. Repository ownership is transferred or mirrored into the buyer's org.
3. Infrastructure is recreated from the repo's configs/operator packet — or
   transferred only where provider terms allow (professional review, R-05).
4. Data (if any exists by then) is migrated via the sealed backup/restore
   and export tooling already in the repo.
5. All keys and credentials are rotated to buyer-generated values.
6. Personal recovery credentials are removed from every transferred surface.
7. Old personal access is revoked (GitHub, providers, billing).
8. Transfer evidence is retained privately (never committed).

## Current findings

- Every account in the inventory is **personally held** (attested) — which
  today is a clean position for a personal-asset sale, but it concentrates
  bus-factor risk (R-03) and guarantees migration work at transfer.
- **Nothing production-grade exists yet to transfer**: no domain, no deploy,
  no data, no brand assets. The dominant near-term transfer surface is the
  repository itself, which transfers cleanly.
- The highest-risk future items are key custody/rotation and any
  data-bearing stores once a real deployment exists — both already have
  in-repo procedures (key rotation tool, sealed backup/restore).
- Provider-account assignability (OpenAI/Anthropic/Notion/Render/
  Cloudflare) remains `professional review required` — plan assumes
  buyer-recreates as the default.
